# Warehouse Rover — Full Project Explanation

This document is a complete, end-to-end explanation of the **Warehouse
Rover Microproject**: the hardware, the wiring, the firmware, every
Python module, every config file, and the math/algorithms that make
it work. It is written so that a reader who has never seen the
project can read it top-to-bottom and understand the whole system.

---

## 1. Project Goal

Build a small **autonomous warehouse robot** that:

1. Sits at a "home" position on a known floorplan.
2. Reads a **QR code** stuck on a package (camera input).
3. Looks up the QR code in a slot dictionary → gets a (x, y) drop point.
4. **Plans a path** through the warehouse using A\* search.
5. **Drives the path** with a differential-drive rover while avoiding
   walls (ultrasonic sensor) and obstacles on the map.
6. **Picks up / drops off** the package with a two-servo gripper.
7. Optionally serves a **live web admin panel** so a human can watch
   the camera, see the planned route, manually drive it, and read
   telemetry.

### High-level architecture

```
                       +-------------------+
   +------+           |  RPi4  (mission)  |          +--------+
   | USB  |  MJPEG -->|  camera, QR, plan,|--> UART->| ESP32  |
   | cam  |           |  steer, odometry  |          |(low lv)|
   +------+           +-------------------+          +---+----+
                                                          |
                                          +---------------+----------------+
                                          |               |                |
                                     L298N motors    2x servos (gripper)   HC-SR04
                                          |               |              MPU6050
                                       DC motors      gripper jaws     dist + yaw
```

* **RPi4 = the brain.** All the perception, planning and high-level
  decisions live in Python.
* **ESP32 = the low-level controller.** It does the things that need
  to be real-time-ish: PWM the motors, drive the servos, time the
  ultrasonic pulse, sample the IMU.
* The two talk over **USB serial** using a tiny, line-based protocol
  described in §7.

## 3. Hardware & Wiring

### 3.1 The two boards

| Board  | Role                                                                 |
| ------ | -------------------------------------------------------------------- |
| RPi 4  | Runs Python. Camera, QR decoding, A\*, navigation, web UI.            |
| ESP32  | Runs Arduino sketch. Motors, servos, ultrasonic, MPU6050, serial.    |

Connected by a **single USB cable** between RPi4 and ESP32. The ESP32
shows up on the RPi4 as a CDC-ACM serial device, typically
`/dev/ttyACM0`, at **115200 baud**.

### 3.2 ESP32 pin map (from `uppdatedPinDiagram.md` — the up-to-date one)

| Component           | Signal | ESP32 Pin |
| ------------------- | -----: | --------: |
| Gripper servo 1     | PWM    | **GPIO18** |
| Gripper servo 2     | PWM    | **GPIO19** |
| Ultrasonic sensor   | TRIG   | **GPIO12** |
| Ultrasonic sensor   | ECHO   | **GPIO13** |
| L298N motor driver  | IN1    | **GPIO25** |
| L298N motor driver  | IN2    | **GPIO33** |
| L298N motor driver  | IN3    | **GPIO32** |
| L298N motor driver  | IN4    | **GPIO27** |
| MPU6050             | SDA    | **GPIO21** |
| MPU6050             | SCL    | **GPIO22** |
| MPU6050             | VCC    | 3.3V      |
| MPU6050             | GND    | GND       |

Reserved for future encoders: **GPIO34** (left), **GPIO35** (right).
Note that GPIO34/35 are **input-only** — they cannot drive outputs.
That is why the motor IN4 went from GPIO34 to GPIO27 in the change
log (`changedPIn.md`).

### 3.3 L298N wiring

```
ESP32 GPIO25 ──> L298N IN1   ┐
ESP32 GPIO33 ──> L298N IN2   ├─ Left side motors  → OUT1 / OUT2
ESP32 GPIO32 ──> L298N IN3   ┐
ESP32 GPIO27 ──> L298N IN4   ├─ Right side motors → OUT3 / OUT4
```

The L298N **ENA / ENB jumpers are assumed installed**. That means the
driver is always enabled and the only way to "stop" a motor is to put
both IN pins LOW. For true PWM speed control later, wire ENA/ENB to
two ESP32 PWM pins and set `MOTOR_LEFT_ENABLE_PIN` /
`MOTOR_RIGHT_ENABLE_PIN` in the firmware (currently `-1` = "jumper
installed, ignore").

### 3.4 Power & ground

Critical — all grounds must be common:

```
ESP32 GND   ─┐
RPi4 GND    ─┤  (through the USB cable)
Servo GND   ─┤
L298N GND   ─┤
MPU6050 GND ─┤
HC-SR04 GND ─┘
```

* Use a **separate 5 V supply** for the servos (a USB charger or
  buck converter is fine). The ESP32's 3.3 V pin cannot power servos
  well — current spikes will brown-out the ESP32.
* Use the **motor battery** (often 7–12 V) for L298N motor power.
* If the HC-SR04 is powered at 5 V, its ECHO pin is 5 V. **Add a
  voltage divider** to GPIO13 (e.g. 1 kΩ + 2 kΩ) — the ESP32 is
  3.3 V tolerant on inputs only up to 3.3 V.
* GPIO12 is an ESP32 **strapping pin**. If the board refuses to boot
  after wiring TRIG to GPIO12, move TRIG to a different safe output
  pin and update the firmware constant.

## 4. Components In Depth

### 4.1 ESP32 (DOIT DevKit V1 or equivalent)

* Dual-core Tensilica LX6, 240 MHz.
* Built-in WiFi/BT (not used here — communication is wired USB).
* 3.3 V logic, 5 V tolerant on **input** only on most pins.
* The Arduino-ESP32 core + the `ESP32Servo` library provide
  `Servo::attach()` and PWM output. The library uses the LEDC
  peripheral internally, so up to 16 simultaneous PWM channels are
  available.
* We use `Wire.begin(SDA, SCL)` to put I²C on **any two GPIO pins**
  (here 21/22) — the ESP32 supports `Wire` on any pin, not just the
  default ones.

### 4.2 L298N dual H-bridge motor driver

* Two H-bridges, one per motor (or per side of the chassis).
* `IN1`/`IN2` direction pins, `ENA` enable / speed pin.
* Logic supply: 5 V. Motor supply: up to 35 V.
* Direction table for one side:

  | IN1 | IN2 | ENA | Motor |
  | :-: | :-: | :-: | :---- |
  |  0  |  0  |  *  | stop (coast) |
  |  0  |  1  |  1  | forward      |
  |  1  |  0  |  1  | reverse      |
  |  1  |  1  |  *  | brake        |

* The firmware implements this in `driveLeftMotor()` /
  `driveRightMotor()`. With ENA jumpered, the speed is effectively
  always "full" and the PWM magnitude is just a "did we go
  forward/backward?" signal.

### 4.3 HC-SR04 ultrasonic distance sensor

* 4 pins: VCC (5 V), TRIG (input), ECHO (output, 5 V), GND.
* The host pulses TRIG HIGH for ≥ 10 µs. The sensor emits a 40 kHz
  burst and raises ECHO HIGH for a duration proportional to the
  round-trip time.
* Distance = `(echo_high_time_us × 0.0343 cm/µs) / 2`.
* `pulseIn(ECHO, HIGH, 30000)` returns 0 on timeout (30 ms = ~5 m
  max). The firmware converts that to `-1.0` so the RPi can
  distinguish "no echo" from "very far".
* The firmware has a hardware safety: if the distance drops below
  `STOP_DISTANCE_CM = 12.0` cm, it **automatically stops the
  motors** even if the RPi hasn't noticed yet.

### 4.4 MPU6050 IMU (6-axis: accelerometer + gyroscope)

* I²C address `0x68`.
* We only use the **Z-axis gyroscope** to integrate yaw.
* On boot we:
  1. Wake the device by writing 0 to register `0x6B` (PWR_MGMT_1).
  2. **Calibrate the gyro bias** by averaging 500 samples while
     stationary.
  3. From then on, in `updateYaw()`, read the raw Z rate
     (register `0x47`), subtract the bias, divide by **131.0 LSB/(°/s)**
     (the ±250 °/s full-scale factor), and integrate over the loop
     period.
* **Caveat:** the MPU6050 is **not a compass**. Yaw drifts
  continuously. For a robot that drives more than a few metres you
  need an external absolute heading source (magnetometer, visual
  fiducials, etc.).

### 4.5 Two hobby servos (the gripper)

* Standard 50 Hz PWM, 1 ms pulse = 0°, 2 ms pulse = 180°.
* The gripper is a two-jaw parallel mechanism. Each servo drives one
  jaw. We **step the angle one degree at a time with 20 ms between
  steps** to avoid jerking the servo gears and to make grasping
  gentler.
* Open / close angles are tunable:
  * Servo 1: 25° (open) → 95° (closed)
  * Servo 2: 155° (open) → 85° (closed)
  (Servo 2's open angle is greater than its close angle because it
  is mounted on the opposite side.)

### 4.6 USB camera

* Any UVC camera. The RPi4 sees it as `/dev/video0` and OpenCV
  opens it by integer index (`cv2.VideoCapture(0)`).
* Used at **640×480** by default — enough for QR detection and not
  heavy on the RPi4's CPU.

### 4.7 Raspberry Pi 4

* 4 GB RAM is plenty. The stack uses ~150 MB of RSS.
* The USB-A port supplies power to the ESP32 and the camera. Use a
  powered hub if the camera is high-current.

### 5.1 `maps/floorplan.png` (200×200 px)

Generated by `create_mock_map.py`. It is a top-down drawing:

```
   0           x grows →         199
 0 ┌──────────────────────────────┐
   │  free  │shelf│ free │shelf│ … │  ← y=40

```

Three vertical "shelves" at x=40..60, x=100..120, x=150..170. The

(mid-2), `x=171..195` (right). All other pixels are white (free).

The generated `occupancy_grid.npy` is a 200×200 uint8 array:
* `0` = free
* `1` = obstacle (or safety-inflated obstacle)

About **39.5 %** of the grid is marked as obstacle after the safety

### 5.2 `maps/occupancy_grid.npy`

Produced by `MapProcessor.generate_occupancy_grid()`. See §6.3 for
the algorithm.

### 5.3 `config/warehouse_slots.yaml`

A dictionary from a **slot ID** to a (x, y) drop point and an
optional (x, y) approach point. The approach point is the cell the
robot aims for first; the drop point is the final cell.

```yaml
home: [20, 20]
slots:
  R1C1:
    drop: [190, 50]
    approach: [180, 50]
  ...
  R3C3:
    drop: [145, 130]
    approach: [140, 130]
```

The convention is image-style: **x grows right, y grows down**.
There are 9 slots, arranged 3 rows × 3 columns of aisles.

## 6. The Python Stack

All Python source lives in `autonomous_rover/src/`. The entry point
is `src/main.py`, which just re-exports `mission_controller.main()`.
The admin panel (`src/admin_panel.py`) is a separate, optional
process.

### 6.1 `src/main.py` — entry point

Tiny shim. It exists so you can run `python src/main.py ...` from
the project root and it forwards to the real CLI in
`mission_controller.py`. The `if __package__ in (None, "")` guard
makes the file work both as a script and as a module of a package.

### 6.2 `src/warehouse_slots.py` — slot config + normaliser

* **`normalize_slot_id(payload)`** — turns any reasonable QR string
  into the canonical `R<row>C<col>` form.
  * Strips, uppercases, removes `-` and `_`.
  * Matches the regex `R0*(\d+)C0*(\d+)`.
  * Returns `f"R{row}C{col}"` with leading zeros stripped.
  * Examples: `"r1c3"` → `"R1C3"`, `"R01C03"` → `"R1C3"`,
    `"r-1_c-3"` → `"R1C3"`. Anything else raises `ValueError`.
* **`SlotDestination`** — frozen dataclass with `slot_id`, `drop`,
  `approach`. The `navigation_target` property returns the
  approach point if it exists, else the drop point. That is the
  point A\* plans to.
* **`WarehouseSlots.load(path)`** — reads the YAML, builds a
  `dict[slot_id → SlotDestination]`, returns a `WarehouseSlots`.
  Also stores `home`.
* **`get_destination(slot_id)`** — looks up by normalised id and
  raises `KeyError` with a helpful message that lists all known
  slots.

#### Why this works

* The normaliser is **forgiving** so QR codes printed at different
  sizes / scanned at angles still decode to the right slot.
* Using a frozen dataclass means the slot table cannot be
  accidentally mutated at runtime.
* Having both `drop` and `approach` lets you plan to a cell that is
  next to the shelf (approach) rather than crashing into the shelf
  (drop). It also makes the path easier to draw — the last segment
  is always a straight "drive in" into the drop cell.

### 6.3 `src/map_processor.py` — image to occupancy grid

`MapProcessor.generate_occupancy_grid(image_path, output_path)` turns
a top-down map image into a 0/1 grid.

Steps:

1. **Load grayscale** with `cv2.imread(..., IMREAD_GRAYSCALE)`.
2. **Gaussian blur** (5×5) to smooth scan noise / paper texture.
3. **Otsu's threshold** with `THRESH_BINARY_INV`: Otsu auto-picks the
   cut-point between the two pixel populations, so we do not have to
   hand-tune "what counts as a wall". The `INV` makes dark pixels
   (walls/shelves) become 255 and light pixels (floor) become 0.
4. **Connected components** filter: tiny blobs (text, single-pixel
   specks) below `min_obstacle_area_px = 40` are dropped. This is the
   "text removal filter" — it stops labels on the floor plan from
   becoming obstacles.
5. **Inflate (dilate) obstacles** by a kernel of radius
   `ceil(robot_radius_cm / resolution_cm_per_px)`. This is the
   **safety margin** — the path planner will keep at least one
   robot-radius away from any wall.
6. **Save** the resulting 0/1 grid as a `.npy` file and return it.

#### Why this works

* Otsu + connected components is a classic, robust way to convert a
  hand-drawn floor plan into a navigable map without ML.
* Dilation is the standard trick to convert "obstacle cells" into
  "cells the robot's centre cannot enter" while still using the same
  A\* code that treats the grid as points.
* Setting `min_obstacle_area_px` is a single knob that controls how
  much detail survives. A noisy photo needs a higher value; a clean
  drawing can use a lower one.

### 6.4 `src/planner.py` — A\* + path pruning

A grid A\* planner with 8-connected neighbours and an Euclidean
heuristic. Two public methods:

* **`plan_path(start, goal) -> list[(x, y)] | None`** — runs A\* and
  returns the dense path (every cell along the way) or `None` if no
  path exists.
* **`prune_path(path) -> list[(x, y)]`** — collapses the dense path
  down to "geometric corners" using line-of-sight.

#### A\* algorithm

* Open set is a binary heap (`heapq`), keyed by `f = g + h`.
* `g_score[node]` is the best known cost from the start.
* `came_from[node]` is the back-pointer for path reconstruction.
* The 8 neighbours have cost 1 (cardinal) or √2 (diagonal). Diagonal
  moves are **only allowed if both side neighbours are also free**
  — that prevents "corner cutting" through walls.
* Heuristic: Euclidean distance. **Admissible** (never overestimates)
  because every step costs ≥ its Euclidean length.
* On reaching the goal, walk `came_from` backwards to reconstruct
  the path, then reverse it.

#### `has_line_of_sight(p1, p2)`

Bresenham's line algorithm. Steps along the line and checks every
cell. If any cell is out of bounds or an obstacle, returns False.
Used by the pruner.

#### `prune_path(path)` — line-of-sight compressor

Greedy. From each waypoint, look at every later waypoint. The first
later waypoint that has line-of-sight becomes the new "current"
waypoint. Repeat.

Concretely, for a path of 500 dense cells you typically end up with
~5–10 corner waypoints. The mission controller then drives to those
corners one at a time, which is much smoother and easier for the
navigator to steer.

#### Why this works

* A\* with an admissible heuristic is **optimal** for the chosen
  cost function. With 8-connected neighbours, the path is geometric
  length-optimal on the grid.
* Bresenham's line algorithm is the standard integer-friendly
  raycast; it visits exactly the cells a straight line passes
  through.
* Pruning by line-of-sight is a standard "any-angle" path
  post-processing trick: A\* is grid-locked, but once you have a
  grid-locked path, you can re-render it as a poly-line of the few
  vertices that are actually needed.

### 6.5 `src/odometry.py` — dead-reckoning pose

`DifferentialOdometry` maintains a `pose = [x, y, theta]` in the map
frame.

Given:
* `wheel_diameter_cm = 6.5`
* `wheel_base_cm = 15.0`
* `ticks_per_revolution = 20`
* `resolution_cm_per_px = 1.0`
* `yaw_sign = 1.0`

It computes `cm_per_tick = π × wheel_diameter / ticks_per_rev`. On
each `update(left_ticks, right_ticks, yaw_deg=None)`:

* `delta_left_cm = (left_ticks − last_left) × cm_per_tick`
* `delta_right_cm = (right_ticks − last_right) × cm_per_tick`
* `distance_cm = (delta_left + delta_right) / 2`
* If `yaw_deg` is given (MPU6050), set `theta = (yaw_deg − yaw_zero) × yaw_sign` (in radians). This **overrides** wheel-derived heading — good for fighting wheel slip.
* Else, derive `delta_theta = (delta_right − delta_left) / wheel_base` and integrate.
* Advance the position: `x += d·cos(θ)`, `y += d·sin(θ)` after
  converting cm → px by dividing by `resolution_cm_per_px`.

#### Why this works

* A differential-drive robot's instantaneous centre of rotation is
  on the line connecting the two wheels. The two-wheel kinematic
  equations are the standard textbook formulas.
* The wheel `cm_per_tick` is the only calibration the user really
  needs to set correctly; everything else is geometry.
* `reset(pose, ...)` lets the mission controller set the initial
  pose to the **home** coordinate from `warehouse_slots.yaml`.

### 6.6 `src/navigator.py` — waypoint follower

`WaypointNavigator` is a **P-controller** that turns
`current_pose = (x, y, θ)` and a `target = (x, y)` into a
`(linear_v, angular_w, arrived)` triple.

```
dx, dy   = target − current
dist     = hypot(dx, dy)
if dist < tolerance:    return 0, 0, True

desired_heading = atan2(dy, dx)
heading_error   = wrap(desired_heading − θ)

if |heading_error| > 0.2 rad (~11°):   # turn in place
    linear_v  = 0
    angular_w = Kp_angular × heading_error
else:                                  # drive forward
    linear_v  = Kp_linear × dist
    if linear_v < min_linear_vel:
        linear_v = min_linear_vel        # creep forward
    angular_w = Kp_angular × heading_error
```

* `Kp_linear = 1.2`, `Kp_angular = 3.5` (from `settings.yaml`).
* `min_linear_vel = 35`, `max_linear_vel = 90`, `max_angular_vel = 90`
  — these are PWM magnitudes in the ESP32's −255..+255 range.

#### `unicycle_to_differential(v, omega)`

Standard differential-drive conversion:
```
left_pwm  = clip(v − ω, −255, 255)
right_pwm = clip(v + ω, −255, 255)
```
A positive `ω` means turn left → the right wheel goes faster than
the left.

#### Why this works

* P-control is the simplest stable steering law. It is not optimal
  but it is robust.
* The 11° "turn-in-place" branch is the key to **non-oscillatory**
  behaviour: instead of trying to drive forward while crabbing, the
  robot stops, faces the target, then drives. This is what
  makes the robot actually converge to a waypoint instead of
  circling it.
* The minimum linear velocity prevents the robot from stalling when
  it is very close to the target but not quite there.

### 6.7 `src/comms.py` — serial link to ESP32

`ESP32Interface` is a thin wrapper over `pyserial`.

* **Open**: `serial.Serial(port, baudrate, timeout=1)`.
* **Simulation mode**: if `port` is `None` / `"NONE"` or `pyserial`
  is missing or open fails, the class falls into
  `simulation_mode = True` and silently **drops every send**. The
  `get_telemetry()` still returns a default snapshot.
* **Send commands** (newline-terminated):
  * `<V,left,right>` — `send_velocity_cmd(l, r)`. Values are
    clipped to ±255.
  * `<G,OPEN>` / `<G,CLOSE>` — `send_gripper_cmd(action)`.
  * `<PING>` — handshake.
  * `<RESET_TICKS>` — zero encoders and yaw.
* **Receive** in a background thread (`_hardware_read_loop`):
  * `<T,distance_cm,left_ticks,right_ticks,yaw_deg>` →
    `Telemetry` dataclass, protected by a lock.
  * `<PONG>` → set `_last_pong` (used by `ping()`).
  * `<READY>` and any other line are consumed (so the RX buffer does
    not fill up) and discarded.

#### Why this works

* The protocol is line-based and ASCII so you can `cat /dev/ttyACM0`
  in a terminal and see exactly what is happening.
* Wrapping the receive loop in a daemon thread means the rest of
  the RPi code can call `get_telemetry()` synchronously without
  ever blocking on `read()`.
* A single `threading.Lock` guards the telemetry struct and
  `_last_pong`. `get_telemetry()` **copies** the dataclass so the
  caller cannot see a half-updated state.
* Simulation mode is the single feature that makes the entire
  Python stack testable on a laptop with no hardware.

### 6.8 `src/qr_scanner.py` — camera → slot ID

`QRScanner.scan(timeout_s=30)`:

1. Open the camera.
2. Loop until timeout: grab a frame, call
   `cv2.QRCodeDetector().detectAndDecode(frame)`.
3. If a non-empty payload comes back, normalise it via
   `normalize_slot_id()` and return.
4. If timeout: raise `TimeoutError`.

`scan_frame(frame)` is the same logic for one frame — used by the
admin panel, which is always reading frames in the background.

#### Why this works

* OpenCV's QR detector is decent, free, and fast enough for 30 Hz
  on the RPi4.
* Wrapping the detector in a normaliser means the rest of the
  codebase only ever sees canonical `R<r>C<c>` strings.

### 6.9 `src/mission_controller.py` — the orchestrator

This is the file that ties everything together. Reading it top to
bottom gives you the whole mission flow.

#### `MissionController.__init__`

* Load `settings.yaml`.
* Open the ESP32 serial link. If the link fails **or** we are in
  `--dry-run`, force `dry_run = True`.
* Load (or regenerate) the occupancy grid.
* Build an `AStarPlanner`.
* Build a `WaypointNavigator` with the P-gains from settings.
* Build a `DifferentialOdometry` with the wheel geometry.
* Load the `WarehouseSlots` table and reset the odometry pose to
  the home coordinate.

#### `run_once()`

```
1.  if not dry_run: open the gripper (so we can pick up a fresh pkg)
2.  slot_id = self._get_slot_id()           # from --qr or QR scan
3.  destination = slots.get_destination(slot_id)
4.  target = destination.navigation_target # approach or drop
5.  print("navigating to", target)
6.  waypoints = self._plan_waypoints(target) # A* + prune
7.  print("planned N waypoints")
8.  if dry_run: return                      # ← we already verified this
9.  gripper CLOSE (pick up the package)
10. send <RESET_TICKS> (zero encoders / yaw)
11. self._follow_waypoints(waypoints)
12. self.esp32.stop()
13. gripper OPEN (deliver)
14. print("Delivery complete.")
```

#### `_get_slot_id()`

* If `--qr` was passed on the CLI, use it.
* Otherwise instantiate `QRScanner` and wait for a real QR.

#### `_plan_waypoints(target)`

* Round the current pose to integer pixels.
* `dense = planner.plan_path(start, target)`. Raise if `None`.
* `pruned = planner.prune_path(dense)`. Drop the first point if it
  equals the start (so we do not re-aim at where we already are).

#### `_follow_waypoints(waypoints)`

* The main 10 Hz control loop. At every iteration:
  1. Read the latest telemetry.
  2. Update odometry (only if `odometry.enabled`).
  3. **Ultrasonic safety**: if `distance_cm > 0 and < obstacle_stop_cm`,
     stop the motors and wait until it climbs back above
     `obstacle_resume_cm`. This handles a real, physical obstacle
     that the map does not know about.
  4. `linear, angular, arrived = navigator.get_steering_commands(pose, target)`.
  5. If `arrived`, pop the waypoint, `stop()` for 0.2 s, continue.
  6. Convert to `(left_pwm, right_pwm)`, clip to `max_drive_speed`,
     send to the ESP32.
  7. `time.sleep(loop_delay)` (default 100 ms = 10 Hz).
* If `odometry.enabled` is **false** (the default and current
  hardware), we are in **dead-reckoning** mode. The RPi sends a
  single timed pulse per cycle and updates its in-software pose from
  the commanded turn/drive, not from encoders.
* If the loop exceeds `max_mission_seconds` (default 180 s) and
  there are still waypoints, raise `TimeoutError`.

#### `_dead_reckoning_step(pose, target)`

Because we cannot read wheel ticks yet, the controller has to *assume*
how fast the robot moves.

* `distance_px = hypot(target − pose)`
* `distance_cm = distance_px × resolution_cm_per_px`
* `forward_time_s = distance_cm / forward_speed_cm_per_s` (capped to
  0.1–0.5 s — the loop will repeat as needed).
* If `|heading_error| > 10°`, **turn in place** for
  `|heading_error| / turn_speed_deg_per_s` seconds, then
  `stop()`. Update the in-software heading by the commanded angle.
* Else drive forward for `step_time_s`, `stop()`, then **update the
  in-software pose** by the distance we just commanded times the
  current heading.

The integration is
```
new_x = x + (PWM/255 × forward_speed_cm_per_s × step_time / res) × cos(θ)
new_y = y + (PWM/255 × forward_speed_cm_per_s × step_time / res) × sin(θ)
```
This works because the PWM duty fraction `(PWM/255)` scales the
maximum speed linearly.

#### Why this works

* The mission controller is intentionally small and linear. Every
  step is named, every step can be run in dry-run, every step
  prints a one-line status. You can `tail -f` a log and watch the
  robot decide.
* The ultrasonic safety check is the **only** hardware sensor
  read on the hot path. Everything else is offline / pre-planned.
  That is what makes the system safe to test with the wheels up.
* The dead-reckoning mode is the **explicitly documented** path for
  the current hardware. The user is told in
  `settings.yaml`:
  > "Set to true only after wheel encoders are wired and the
  > ESP32 is reading them."

### 6.10 `src/admin_panel.py` — the web admin panel

A single-file Python HTTP server using only the standard library
(`http.server.ThreadingHTTPServer`).

#### Architecture

```
                  ┌──────────────────────────────────────┐
   Browser   ──>  │  AdminRequestHandler (BaseHTTP…)     │
                  │     │                                │
                  │     ├── GET  /            -> HTML    │
                  │     ├── GET  /video       -> MJPEG   │
                  │     ├── GET  /api/status  -> JSON    │
                  │     ├── GET  /api/snapshot.jpg       │
                  │     ├── GET  /api/map.jpg            │
                  │     └── POST /api/command?action=…   │
                  │                                      │
                  │  AdminState (singleton-ish)          │
                  │     ├── CameraWorker  (thread)       │
                  │     ├── ESP32Interface (comms)       │
                  │     ├── AStarPlanner                 │
                  │     ├── WarehouseSlots               │
                  │     └── occupancy grid (numpy)       │
                  └──────────────────────────────────────┘
```

* **`CameraWorker`** — a daemon thread that loops at ~30 Hz, reads
  the camera, runs `cv2.QRCodeDetector().detectAndDecode`, draws a
  green polygon around the QR, encodes as JPEG, stores the latest
  bytes and the latest slot ID under a `threading.Lock`.
* **`AdminState`** — owns the camera, the ESP32, the planner, the
  grid, the current `pose`, the `logs` deque (capped at 120).
* **Routes**:
  * `/` — single-page HTML app with `<img src="/video">` and a JS
    poll loop hitting `/api/status` once a second and refreshing
    `/api/map.jpg?t=…` at the same time.
  * `/video` — `multipart/x-mixed-replace` MJPEG. The handler
    loops forever, writing the latest JPEG from `CameraWorker`.
    The browser shows it as a live stream.
  * `/api/status` — JSON with ESP32 connection, camera OK flag,
    latest QR, destination, route, telemetry, log tail, known
    slots, manual-busy flag, last command.
  * `/api/map.jpg` — the occupancy grid with the current pose, all
    slot drop points, home, and the planned route (if any) overlaid
    in colour. Cached in the browser via `?t=…`.
  * `/api/snapshot.jpg` — single-shot camera frame.
  * `/api/command` — accepts a `POST` with `?action=forward|backward|
    left|right|stop|gripper_open|gripper_close|ping`. Manual nudges
    run in a daemon thread: send the velocity, sleep for
    `nudge_duration_s` (default 0.35 s), `stop()`. The
    `manual_lock` ensures only one nudge runs at a time.

#### Why this works

* Pure-stdlib HTTP server — no Flask, no Tornado, no extra deps on
  the RPi. The whole admin panel adds only `cv2` and `PyYAML` (which
  the rest of the system already needs).
* MJPEG over `multipart/x-mixed-replace` is the simplest way to
  stream live video to a browser with no JS and no codec
  negotiation. Modern browsers all still support it.
* The map preview is a regular JPEG that the JS just refreshes with
  a cache-busting query string — no need for canvas / WebGL.
* The manual controls are throttled server-side so a user mashing
  the "Forward" button cannot queue up multiple overlapping nudges.

## 7. The Serial Protocol

```
RPi4 → ESP32  (newline-terminated, ASCII):
    <V,left,right>   signed motor PWM, range −255..+255
    <G,OPEN>         open gripper
    <G,CLOSE>        close gripper
    <PING>           handshake
    <RESET_TICKS>    zero encoder counters and yaw

ESP32 → RPi4:
    <T,distance_cm,left_ticks,right_ticks,yaw_deg>
    <PONG>
    <READY>            sent once on boot
```

* Commands and replies are bracketed by `<` `>` so a partial line
  (e.g. someone `cat`-ing the port in another window) cannot be
  misread as a complete command.
* Telemetry is sent at 10 Hz (`TELEMETRY_INTERVAL_MS = 100`).
* Encoder snapshots are taken inside a `noInterrupts()`/`interrupts()`
  pair so they are atomic with respect to the encoder ISRs.

## 8. End-to-End Mission Flow

Here is what happens when you run `python src/main.py --qr R1C1`:

1. **`main.py` → `mission_controller.main()`**
2. `MissionController.__init__`:
   * Loads `settings.yaml`. Sets `odometry_enabled = False`.
   * Opens `/dev/ttyACM0` at 115200 baud. **Success** here is what
     printed `[COMMS] Real ESP32 connection established on port: /dev/ttyACM0`.
   * Loads `maps/occupancy_grid.npy` (200×200).
   * Creates `AStarPlanner`, `WaypointNavigator`, `DifferentialOdometry`,
     `WarehouseSlots`. Resets odometry to `(20, 20, 0)`.
3. `run_once()`:
   * Since `--dry-run` is not set, opens the gripper.
   * `--qr R1C1` → no camera needed. `slot_id = "R1C1"`.
   * `slots.get_destination("R1C1")` →
     `SlotDestination(slot_id="R1C1", drop=(190, 50), approach=(180, 50))`.
   * `target = (180, 50)` (approach point, not drop).
   * `plan_path((20, 20), (180, 50))` → returns a list of cells like
     `[(20,20), (21,20), (21,21), …, (180,50)]`. **3 waypoints** after
     pruning: `[(170, 35), (175, 38), (180, 50)]` for the mock map.
4. **In a real run (not dry):**
   * `gripper CLOSE` for 0.8 s.
   * `send <RESET_TICKS>`.
   * 10 Hz loop until the waypoint list is empty or 180 s elapses:
     * Get telemetry, update pose, run the safety check.
     * Compute `(v, w)`, convert to `(L, R)` PWM, clip, send.
   * `stop()`.
   * `gripper OPEN` for 0.8 s.
   * `print("[MISSION] Delivery complete.")`.

## 9. The Admin-Panel Mission Flow

The admin panel is a **non-driving** UI by default. It shows:

* Live camera (with QR overlay).
* The planned route to the **currently detected** QR's slot.
* Manual nudge buttons. Each press spawns a one-off thread that
  drives for `nudge_duration_s` (default 0.35 s) then stops. The
  "pose" shown in the map is not updated by these nudges — the
  admin panel does not run a localizer. It is a *tether*, not an
  autonomous driver.

## 11. Algorithms Recap (math)

### A\* heuristic
```
h(a, b) = √((a.x − b.x)² + (a.y − b.y)²)
```
Admissible (≤ true cost on the grid), consistent, gives an optimal
8-connected path.

### Bresenham line-of-sight
```
dx, dy  = abs(b − a)
sx, sy  = sign(b − a)
err     = dx − dy  (or dx/2, dy/2 depending on variant)
while not at b:
    if grid[y][x] is obstacle:  return False
    err2 = err
    if err2 > -dx:  err -= dy;  x += sx
    if err2 <  dy:  err += dx;  y += sy
return True
```

### Unicycle → differential drive
```
L = v − ω
R = v + ω
clip both to ±max_pwm
```
(`ω > 0` = turn left.)

### Differential-drive forward kinematics
```
v       = (v_R + v_L) / 2
ω       = (v_R − v_L) / wheel_base
x'      = x + v·cos(θ)·dt
y'      = y + v·sin(θ)·dt
θ'      = θ + ω·dt
```

### Wheel ticks → cm
```
cm_per_tick = π · wheel_diameter / ticks_per_revolution
delta_cm    = (ticks_now − ticks_last) · cm_per_tick
```

### Gyro → yaw
```
gyro_z_dps  = (raw_z − bias) / 131.0   // 131 LSB/(°/s) at ±250°/s
yaw_deg    += gyro_z_dps · dt          // integrate
```

### Ultrasonic → cm
```
distance_cm = (echo_high_us · 0.0343) / 2
```

## 13. How To Use It

### 13.1 One-time setup

```bash
cd autonomous_rover
bash setup.sh
```

What `setup.sh` does:

1. Create `.venv/` with the system `python3`.
2. `pip install -r requirements.txt` (numpy, opencv-contrib-python,
   matplotlib, pyserial, PyYAML, qrcode, Pillow).
3. Add the current user to the `dialout` group (so `pyserial` can
   open `/dev/ttyACM0`). If sudo asks for a password, type it,
   then **log out and log back in** so the new group is active.
4. `chmod a+rw /dev/ttyACM0` if the port exists.
5. Regenerate `maps/occupancy_grid.npy` from `maps/floorplan.png`.
6. Generate one QR PNG per slot ID in `qrcodes/`.

### 13.2 ESP32 firmware

1. Open `esp32/rover_firmware/rover_firmware.ino` in Arduino IDE.
2. Install ESP32 board support + `ESP32Servo` library.
3. Select the right port, click **Upload**.
4. Open Serial Monitor at 115200 baud. You should see `<READY>`
   within a couple of seconds.

### 13.3 Run a mission

```bash
.venv/bin/python src/main.py --qr R1C1 --dry-run   # plan only
.venv/bin/python src/main.py --qr R1C1            # actually drive
.venv/bin/python src/main.py                     # scan QR then drive
```

### 13.4 Run the admin panel

```bash
.venv/bin/python src/admin_panel.py
# open http://192.168.0.11:8080
```

The browser shows live camera, map+route, telemetry cards, and the
manual controls.

### 13.5 Generate QR codes

```bash
.venv/bin/python generate_qr_codes.py
# qrcodes/R1C1.png ... R3C3.png
```

Print them and tape them to the packages. The RPi4 USB camera will
pick them up automatically.

## 14. Why Everything Works (end-to-end summary)

* **The map is one consistent frame.** All coordinates —
  warehouse slots, planner cells, odometry pose, mission target —
  are in the same `(x, y)` pixel frame as `floorplan.png`. The
  `resolution_cm_per_px` is the only place that maps pixels to real
  distances.
* **A\* is the source of truth for "where can I go?"** Every other
  module trusts the grid: the planner's pruner, the mission
  controller, the admin panel's route preview.
* **The navigator + dead-reckoning step form a closed loop** that
  always steers toward the next waypoint. Even when odometry is
  off, the visualised route in the admin panel and the printouts
  in dry-run let you *see* what the robot intends to do.
* **The serial protocol is line-based and ASCII.** You can `cat
  /dev/ttyACM0` in another shell and watch the telemetry stream.
* **The simulation mode in `comms.py` is the cornerstone of
  testability.** Every code path that talks to the ESP32 has a
  silent no-op fallback, so the entire Python stack runs on a
  laptop with zero hardware. That is why `run_tests.py` works
  everywhere.
* **The admin panel and the mission are decoupled.** The admin
  panel never starts a mission on its own; it only shows status
  and sends manual nudges. This makes it safe to leave running on
  the RPi4 while you iterate on the mission code.

| Limitation                                | Why                                                | Fix                                           |
| ----------------------------------------- | -------------------------------------------------- | --------------------------------------------- |
| No wheel encoders wired                   | Current hardware                                   | Add hall-effect encoders, set `odometry.enabled: true` |

## 17. TL;DR

* The **RPi4** is the brain. It owns the camera, the QR, the A\*
  path, the navigator, and the web UI. It tells the **ESP32** what
  to do over USB serial.
* The **ESP32** is the low-level controller. It owns the motors,
  the gripper, the ultrasonic, and the IMU. It streams telemetry
  back to the RPi4 ten times a second.
* The **map** is a single image (`floorplan.png`) turned into a
  0/1 grid. **A\*** finds a path on that grid, the pruner shrinks
  the path to corner waypoints, the **navigator** drives to each
  waypoint, and the **ultrasonic** acts as a last-line-of-defence
  stop.
* **No encoders** are currently wired, so the navigator runs in
  **dead-reckoning** mode: it sends a single timed pulse per loop
  and integrates the assumed forward/turn rate in software.
* The **admin panel** is a self-contained `http.server` that
  streams MJPEG and exposes JSON status + manual nudges. It never
  starts a mission on its own.
* **Tests** run end-to-end with no hardware thanks to
  `simulation_mode` in `comms.py` and the `--dry-run` flag in
  `mission_controller`. 16/16 pass on a clean checkout.
